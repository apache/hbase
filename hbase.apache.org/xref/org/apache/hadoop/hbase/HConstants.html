302   // Always store the location of the root table's HRegion.
303   // This HRegion is never split.
304 
305   // region name = table + startkey + regionid. This is the row key.
306   // each row in the root and meta tables describes exactly 1 region
307   // Do we ever need to know all the information that we are storing?
308 
309   // Note that the name of the root table starts with "-" and the name of the
310   // meta table starts with "." Why? it's a trick. It turns out that when we
311   // store region names in memory, we use a SortedMap. Since "-" sorts before
312   // "." (and since no other table name can start with either of these
313   // characters, the root region will always be the first entry in such a Map,
314   // followed by all the meta regions (which will be ordered by their starting
315   // row key as well), followed by all user tables. So when the Master is
316   // choosing regions to assign, it will always choose the root region first,
317   // followed by the meta regions, followed by user regions. Since the root
318   // and meta regions always need to be on-line, this ensures that they will
319   // be the first to be reassigned if the server(s) they are being served by
320   // should go down.
321

323   public static final byte [] ROOT_TABLE_NAME = toBytes("-ROOT-");
